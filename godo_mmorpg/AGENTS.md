# MMORPG 服务端开发约定

## 适用范围

- 本文件用于基于 KBEngine-Nex 开发 MMORPG 服务端游戏资产库。
- 当前文件暂放在 KBEngine-Nex 引擎仓库根目录；生成独立服务端资产库后，应将它移动到该资产库根目录。
- 客户端位于独立仓库，不在本仓库中修改。本服务端可以对接 Godot、Unity、Unreal、Cocos、自研客户端等，不把业务逻辑绑定到某个客户端引擎或 SDK 语言。
- 客户端协议需要变化时，以服务端实体定义为唯一协议源，按实际目标导出 Unity、UE5、C++、C#、TypeScript 等 SDK，并明确列出各客户端仓库需要配合的事项。
- 默认只开发游戏逻辑，不修改 `kbe/src/`、`kbe/res/` 等引擎源码。只有需求明确指向引擎缺陷，并且已有最小复现时，才允许修改引擎。

## 开始任务前

1. 先定位相关实体、`.def`、公共类型、配置和既有测试，说明当前行为以及期望行为。
2. 给出可验证的完成标准。Bug 必须先有稳定复现；新功能至少覆盖正常路径、非法输入和重复请求。
3. 存在两种以上合理实现时，列出最小方案及取舍，等待确认后再编码。
4. 不确定 KBEngine API、回调签名、属性 Flags 或生命周期时，查本仓库文档与源码，不凭记忆编写。
5. 发现需求与实体归属、数据权威或现有协议冲突时，立即停止并说明冲突。

## 全生命周期原则

- 项目按立项/架构、开发、测试、上线、运营维护、停运六个阶段治理；单项需求至少覆盖开发、测试、上线和维护影响。“代码已经写完”不等于交付完成。
- 在开始编码时就定义配置、可观测性、兼容性、发布与回滚方案，不到上线前临时补齐。
- 开发、测试、预发布和生产环境必须隔离数据库、账号、密钥、日志与外部服务。禁止让本地或测试进程连接生产资源。
- 构建产物、游戏脚本、配置数据、数据库 schema 和客户端协议分别版本化，并能追溯到同一发布版本。
- 自动化流程尚未建立时，保留人工检查清单并记录执行证据；不得假装存在 CI/CD、灰度平台或监控能力。

## 成熟度与实施优先级

- 所有阶段都必须遵守：服务端权威、资产不重复/不凭空丢失、持久化结果不伪报、安全凭据不泄露、队列与输入有界。
- 学习/原型阶段优先完成单区最小闭环和自动化测试，不提前建设跨区、微服务、多活或复杂运营平台。只要服务暴露到互联网，网络隔离、强凭据、限流、备份和日志脱敏仍是硬要求。
- 内测阶段必须具备可重复部署、协议版本管理、基础监控告警、备份恢复、Bots 压测、GM 审计和故障处理清单。
- 公测/商业运营前必须具备明确 SLO、RPO/RTO、容量与成本模型、值班响应、灰度回滚、灾备演练、支付对账、隐私治理和停运预案。
- 规则按风险实施，不为尚不存在的规模过度设计；但任何延期的生产能力必须记录负责人、完成条件和最迟启用阶段。

## 立项与架构阶段

- 先定义核心玩法循环、目标平台/地区、预计 CCU/DAU、登录峰值、同屏密度、Tick/同步频率、社交规模、数据保存周期和商业化方式，再选择拓扑与数据模型。
- 画出服务端权威图：账号、角色、Space、战斗、背包、经济、公会、聊天、匹配、排行榜、活动、支付分别由谁持有和写入，失败时谁负责恢复或补偿。
- 明确区服/大区/Region/Shard/Space 的边界、玩家路由、唯一 ID、名称唯一性和跨区需求。没有真实规模需求时采用最简单的单区方案，但接口和数据不得暗含错误的全局唯一假设。
- 为登录、角色加载、进入世界、战斗 RPC、写库和维护恢复定义 SLI/SLO；同时定义可接受数据丢失量 RPO 与恢复时间 RTO。目标必须可测，不能只写“高可用”“低延迟”。
- 列出数据库、身份、支付、邮件、对象存储、监控等依赖及超时/降级策略，确认每个依赖故障不会造成资产重复、主循环阻塞或全服雪崩。
- 建立容量与成本假设：单进程可承载 Entity/玩家、CPU/内存/带宽、数据库 IOPS/存储增长、日志与备份成本；在真实压测后修正，不拿宣传值当容量结论。
- 对关键架构决定记录简短 ADR：问题、约束、候选方案、选择理由、代价和复审条件。不要用复杂文档替代代码，但不能让关键一致性设计只存在于口头讨论。
- 每个核心系统明确代码责任人、数据责任人、告警负责人和运行手册；个人项目也应让未来的自己能在事故中按文档恢复。

## 资产库结构与职责

服务端游戏资产库遵循 KBEngine-Nex 生成的结构：

- `entities.xml`：注册实体及其 Base、Cell、Client 组成。
- `entity_defs/`：实体属性、RPC 和共享数据类型，是服务端与客户端的协议源。
- `base/`：账户、角色、背包、装备、任务、社交、交易、邮件、公会、经济和持久化协调。
- `cell/`：Space、AOI、移动、战斗、技能、AI、掉落与场景内临时状态。
- `common/`：无组件状态的共享常量、纯函数和数据定义；不得在导入时产生业务副作用。
- `login/`：认证入口和登录策略，不承载角色业务。
- `interface/`：受控的外部服务接入；所有响应均按不可信输入处理。
- `bots/`：协议回归、在线压力和玩法自动化验证。
- `res/server/`：环境配置。密钥、密码和生产地址不得提交到 Git。
- `plugins/`：仅用于边界清楚、确实需要复用或多人独立维护的模块。一次性功能不为“将来可能复用”而插件化。

新增玩法优先完成一个可运行的纵向切片，再拆分系统。不要一开始同时铺开背包、任务、公会、交易等多个未闭环模块。

## 服务端权威边界

- 客户端永远是不可信的表现层。客户端只能表达意图，不能提交最终位置、伤害、掉落、经验、价格、货币余额、冷却结果或任务完成结果。
- Base 实体是持久状态和跨场景业务的权威；Cell 实体是当前 Space 内实时状态的权威。
- 移动合法性、目标可见性、距离、阵营、技能消耗、冷却和命中由 Cell 校验与结算；需要持久化的最终奖励交由 Base 应用。
- 背包、货币、装备、邮件附件、拍卖和玩家交易必须由 Base 端以单一权威入口修改。禁止从多个系统直接改同一份余额或物品集合。
- 跨实体操作必须定义所有权、请求方、成功回执、超时、重试与补偿策略。不能依赖 RPC 必然到达或固定顺序到达。
- 随机数在服务端生成。影响经济或战斗结果的随机过程应能记录关键输入，便于审计和复现。

## Entity 与协议设计

- Entity 表示具有身份、生命周期或分布式位置的对象；普通值对象和纯计算不要滥建 Entity。
- 每个新 Entity 必须先说明：是否有 Base、Cell、Client，谁创建和销毁，是否持久化，断线和进程故障后如何恢复。
- Entity 属性只保存同步、迁移或持久化确实需要的状态；高频临时计算值保留在脚本对象中，避免无意义的网络广播和写库。
- RPC 名称表达业务意图。参数使用最小必要类型，不用 `PYTHON` 承载面向客户端或跨组件的任意对象。
- 修改 `entities.xml`、`.def`、`types.xml`、属性 Flags、RPC 参数或持久化结构属于协议/schema 变更：
  1. 明确兼容性与存量数据迁移方案；
  2. 更新服务端测试；
  3. 使用资产库 `scripts/gensdk.bat`、`gensdk.sh` 或 `kbcmd --clientsdk=<target>` 重新生成所需客户端 SDK；
  4. 通知所有受影响的客户端仓库同步生成物；
  5. 完整重启相关组件，不把它当作 Python 逻辑热更。
- SDK 生成物不得手工修改。生成后记录目标、服务端脚本版本和协议变更摘要，并验证生成命令成功、目标文件完整。
- 避免超大 `Avatar` 类。按明确业务边界拆成组件、辅助对象或插件，但保持调用链可追踪，避免循环依赖。

## 分布式拓扑与状态所有权

- 任一关键状态在同一时刻只能有一个权威写入者。使用明确的 Entity 所有权、版本/世代或等价防陈旧机制，防止旧回调、旧进程或重复角色实例继续写入。
- 登录、选角、上线、进入 Space、传送、跨区、断线、顶号、下线和销毁使用显式状态机；列出允许转换、超时和失败恢复，禁止用零散布尔值组合出不可解释状态。
- 角色传送或跨服迁移必须定义源端冻结、目标端接管、提交确认和失败补偿。任何时刻都不能出现两个可操作角色，也不能两边都认为对方已接管。
- 跨 Entity/区服事务不假设全局 ACID。采用幂等命令、唯一业务号、状态机/补偿和最终核对；确实要求强一致时集中到单一权威写入者并评估热点。
- 不依赖跨进程消息的全局顺序。需要顺序的业务携带版本、序列或前置状态，并处理重复、过期、丢失和乱序消息。
- 全服排行榜、公会、拍卖、世界事件等全局系统先定义分片键、聚合周期、一致性目标和热点上限；不要把所有请求集中到一个永不迁移的 Python 对象。
- 多 Region 部署需明确延迟、数据驻留、账号归属、跨区社交和故障切换语义。未实际验证前不宣称跨地域双活。

## MMORPG 核心业务规则

### 战斗与移动

- 战斗结算采用清晰且固定的顺序，例如：前置校验、资源消耗、命中、伤害/治疗、状态效果、死亡、奖励、事件通知。
- 所有数值明确单位、范围、取整和溢出策略；时间统一使用服务端时间，持续时长优先使用单调时钟语义。
- 高频 Tick 中禁止同步数据库、阻塞 I/O、大对象复制和无界遍历。AI、AOI 与广播需要有预算，并通过压测验证。
- 断线重连、切换 Space、传送、死亡和下线过程中，明确角色在哪个阶段可交互，防止重复实体和幽灵状态。

### 物品与经济

- 货币使用整数最小单位，不使用浮点数。
- 所有资产变化使用统一服务入口，并记录原因、来源、目标、数量、业务流水号和结果。
- 发奖、充值回调、邮件领取、任务提交、合成、购买和交易必须幂等；重复请求不得重复增加或扣除资产。
- 多步交易先验证全部前置条件，再提交状态变化；失败必须无部分成果，或有明确、可重复执行的补偿流程。
- 对客户端提交的物品 ID、数量、槽位、价格和归属逐项校验，并设置合理上限。
- 为所有货币与高价值物品建立可追溯账本或等价不可抵赖流水。余额是账本结果/快照，不能只有一个可随意覆盖的最终数字。
- 每种资产明确来源与消耗口，持续监控产销比、存量、价格、集中度和异常转移。数值正确但导致通胀失控同样属于服务端事故。
- 定期执行资产守恒核对：业务订单、账本流水、角色余额和外部平台记录能对应；差异进入隔离队列人工处理，不自动重复发奖。

### 支付与商业化

- 支付订单使用不可逆唯一订单号和明确状态机，覆盖创建、支付中、到账、发货、失败、超时、退款和拒付；状态转换幂等且可审计。
- 不信任客户端提交的商品、价格、币种或支付结果。以服务端商品配置和经过签名验证的平台回调/主动查询为准，并校验金额、币种、账号和订单状态。
- 支付成功与游戏发货分离但最终可核对；回调重复、乱序、延迟或服务重启不能重复发货。无法确认结果时保持待处理，不猜测成功。
- 退款、拒付、补单和人工补偿使用独立受控流程，保留原订单关联、审批、前后资产和操作者，禁止直接改余额掩盖问题。

### 活动、GM 与社区治理

- 活动配置不可变版本化，明确目标区服、开始/结束时区、资格、奖励上限、补发和回滚。上线前支持离线校验、预览/演练和小范围灰度。
- 运营配置变更和批量任务具有审批、影响人数预估、限速、暂停、续跑、幂等、结果汇总与审计；大批量操作不能在主循环一次完成。
- GM 工具采用 RBAC。发放、封禁、回档、传送、改名等高风险动作支持原因、工单号、二次确认和范围上限，禁止提供任意 Python/SQL 作为日常运营入口。
- 聊天、命名、公会公告和玩家生成内容具备频率限制、举报、屏蔽、禁言和审核证据；处理 Unicode 混淆、控制字符、垃圾信息与隐私泄露。
- 封禁与处罚区分账号、角色、设备和聊天等范围，记录证据、期限与申诉结果；反作弊检测、处罚策略和客服解释相互解耦。

### 任务、社交与世界状态

- 任务进度由已验证的服务端领域事件推进，不接受客户端直接设置进度或完成状态。
- 好友、公会、组队、聊天等跨玩家操作必须处理目标离线、重复操作、并发修改和权限变化。
- 世界唯一状态要有唯一权威实体或明确的分片规则；不要把需要强一致的数据随意存入 `globalData`、`baseAppData` 或 `cellAppData`。

## 持久化与一致性

- 明确每个持久字段的写入时机：关键资产变化及时落库，普通进度可合并写入，但必须定义最大可接受丢失窗口。
- `writeToDB` 等异步操作必须处理回调结果、实体销毁和失败重试；禁止把“已发起写入”视为“已经持久化”。
- 数据迁移使用可重复执行、可审计、可回滚或可补偿的独立流程，不放进 `onReload`、导入副作用或普通登录回调中偷偷执行。
- schema 演进优先采用“扩展—迁移—切换—清理”：先增加兼容结构，再迁移/核对，切换读写后最后删除旧结构。临时双读/双写必须有一致性检测和明确下线日期。
- 每次迁移记录 schema 版本、数据范围、批次游标、成功/失败数和校验结果；可安全暂停和续跑，不能因进程重启从头重复破坏数据。
- 数据按公开、内部、敏感、关键资产分类，分别定义访问权限、加密、日志脱敏、备份、保留和删除策略。生产数据进入测试环境前必须脱敏。
- 玩家删除账号、地区数据驻留、未成年人和支付记录等要求按实际发行地区确认；法律与平台规则不明确时先记录风险并咨询专业人员，不凭代码假设合规。
- 服务安全关闭使用资产库 `scripts/safe_kill.bat` 或 `safe_kill.sh`。生产环境不得使用强制结束进程代替安全关闭。
- 备份恢复、断线重连和组件异常恢复都要用真实流程演练，不能只验证正常启动。

## 安全要求

### 网络与基础设施

- 先画清攻击面：客户端只应访问明确需要的 LoginApp/BaseApp 外部地址与端口。Machine、Logger、DBMgr、BaseAppMgr、CellAppMgr、CellApp、数据库及内部通信端口必须处于受控内网。
- Telnet、Watcher、WebConsole、Python Console、管理后台和调试端口不得暴露到公网；通过防火墙/安全组、来源白名单、强认证和最小权限限制访问，并记录管理操作。
- `kbengine_defaults.xml` 和生成模板中的密码、证书、账号仅是示例，生产必须在资产库配置或密钥系统中全部覆盖。不同环境不得复用密钥，私钥不得进入 Git、镜像或日志，并建立轮换与吊销流程。
- 外部登录与游戏通道启用适合目标 SDK 的加密方案；跨不可信网络访问数据库或外部服务时使用 TLS 并校验证书。不得因联调方便永久关闭证书校验。
- 只开放实际使用的 TCP/UDP 端口和协议。登录、注册、下载、聊天等公开入口需在边缘层和业务层同时限流；大规模 DDoS 依赖网络/云防护，不能指望 Python 业务逻辑单独抵御。
- 外部通道的消息数、字节数、发送窗口和接收窗口必须有经过压测的有限阈值。不得为了“避免掉线”把所有溢出限制设为无限。

### 身份、会话与权限

- 密码、验证码、会话 Token、私钥和完整外部签名永不写日志。生成模板中的调试输出也必须检查，不能原样带入生产。
- 账号密码使用成熟的慢哈希与独立盐，或交给受信任身份服务处理；不得自创加密算法、可逆保存明文密码或仅依赖传输加密。
- 会话具有过期、撤销、重复登录和异常断开策略。重连必须重新确认会话与角色归属，不能只凭客户端提交的 dbid/entityID 恢复身份。
- 权限默认拒绝。玩家、客服、运营、GM、自动任务和数据库迁移使用不同身份与权限；高风险管理操作需要复核或双人审批。
- 账号注册、登录、找回密码、兑换码和验证码接口防止枚举、撞库、暴力破解与重放；对外错误信息不泄露账号是否存在、内部堆栈和数据库细节。

### RPC、业务与反作弊

- 每个客户端 RPC 都校验：调用者状态、权限、参数类型与范围、资源归属、目标可见性、距离、冷却和请求频率。
- 登录、聊天、交易、邮件、拍卖、角色命名等入口实施限流和长度限制；文本经过规范化并防止控制字符、注入和日志污染。
- 对请求增加合理的序列号、时间窗口或业务流水号；涉及资产与奖励的接口必须同时防重复、重放和并发竞态。
- 移动反作弊结合服务端时间、速度/加速度、碰撞、可达性、传送状态和网络抖动判断；不能简单相信坐标，也不能用零容忍规则误伤高延迟玩家。
- 战斗反作弊校验施法频率、技能状态、资源、距离、视野、目标状态和事件顺序。客户端动画结束、命中特效或伤害数字不是结算证据。
- 对异常移动、战斗、经济和社交行为记录可解释的证据与风险分，不在单一偶发信号上永久封禁。自动处置必须可审计、可申诉和可撤销。
- 外部回调必须校验签名、时间窗口、重放和订单状态。日志不得输出密码、Token、完整会话密钥或其他敏感数据。
- 管理命令与普通玩家 RPC 分离，执行时记录操作者、目标、原因、前后值和结果。
- 原始 SQL/数据库命令应尽量禁止。数据库命令黑名单只能作为纵深防御，不能替代参数化访问、数据库最小权限和代码审查；迁移账号与运行账号分离。
- 文件路径、资源下载、HTTP 回调与外部 URL 必须限制允许的协议、主机、大小和超时，防止路径穿越、SSRF、恶意大响应及无限等待。
- 新增依赖或构建产物要确认来源、许可证、校验值与已知漏洞；固定可重复安装的版本，不直接执行来历不明的脚本或二进制。
- 按最小必要范围收集玩家数据，定义访问、保留和删除规则；导出的日志、备份和测试数据同样按敏感数据保护。
- 任何绕过校验、临时发奖或调试后门都不得进入提交。

## 性能与容量约束

### 主循环与计算预算

- BaseApp/CellApp 主线程中禁止阻塞数据库、文件、网络或子进程调用。所有外部 I/O 使用有超时的异步接口，完成回调只做有界工作。
- 为服务器 Tick、关键 RPC、AI、战斗和序列化定义可测预算，关注 P50/P95/P99 和最坏情况。没有目标负载与预算的“优化”不算完成。
- 单次 Tick、Timer 或 RPC 不得遍历无界玩家/实体/物品集合。大任务拆成有进度、有上限、可取消的批次，并在多个 Tick 间公平调度。
- 不在高频路径创建大量临时字典/列表、深拷贝完整状态、拼接大日志或重复解析配置。先用 `cprofile`、`pyprofile`、`eventprofile`、`networkprofile` 或等价测量定位，再优化热点。
- Timer 数量和触发频率必须有预算。大量同类倒计时优先使用分桶、时间轮、统一调度或按需计算，不为每件物品、Buff 或任务盲目创建高频 Timer。
- AI、寻路和感知仅在必要时运行。利用 AOI/Witness 与 `onWitnessed` 等生命周期降低无人观察对象的更新频率或休眠，但不能改变应持续推进的权威规则。

### 网络与分布式调用

- 客户端同步采用兴趣管理、差量更新、合并与限频；不向无关玩家广播，不每 Tick 重发未变化属性，不把大型快照塞入单个 RPC。
- 为每个高频 RPC 记录调用频率、典型/最大负载和接收方数量。区分关键可靠消息与可丢弃状态更新，按玩法需要选择 TCP/UDP，不能凭习惯决定。
- Base/Cell 及跨 Entity 调用避免聊天式往返和 N+1 RPC。优先传递最小业务结果，批量读取/写入，但批次必须有大小上限。
- 序列化类型优先使用明确的 `.def` 类型；避免用 `PYTHON` 传递体积不可控对象。字符串、数组、字典、聊天内容和自定义二进制均设置长度上限。
- 监控外部通道溢出、排队字节、丢包/重传、断连和每玩家带宽。慢客户端不能无限积压发送队列拖垮进程，应合并旧状态或安全断开。
- 组件负载均衡不能替代业务分片。世界 Boss、拍卖、全服排行榜和公会等热点需要设计分片、聚合或串行权威入口，避免单个全局 Entity 成为瓶颈。

### 数据库、缓存与内存

- 数据库访问避免 N+1 查询、全表扫描、无索引排序和超大事务。新增查询要说明数据规模与索引，使用执行计划或压测验证，不凭小数据集判断。
- 合并非关键的频繁写入并设置最大落库延迟；关键资产仍以正确性优先。不能为提高吞吐而让货币、交易或充值结果只存在内存中。
- 缓存必须有容量上限、淘汰和一致性策略。缓存不是权威数据，不允许用永久进程内字典保存所有离线玩家或无界历史记录。
- Entity、组件、Timer、回调、邮件、聊天、排行榜和审计队列都要有生命周期与上限。断线、离开 Space、任务取消和实体销毁时释放引用，防止隐性内存泄漏。
- 数据库连接数、线程池大小、备份周期、归档周期和 Entity 恢复批次基于实测调整；修改默认值时记录原因、负载模型和回退值。
- 容量评估至少覆盖 CCU、登录峰值、同屏实体、每秒 RPC、每秒数据库读写、网络带宽、内存/Entity、日志增长与备份窗口，并预留故障和活动峰值余量。

## 稳定性与过载保护

- 所有跨组件 RPC、数据库写入、HTTP/第三方调用和后台任务都定义超时、失败结果和调用方行为。禁止无限等待或悄悄吞掉失败。
- 只有幂等操作可以自动重试；重试采用有上限的指数退避与抖动，避免组件恢复时形成重试风暴。非幂等操作使用业务流水查询最终状态。
- 队列、批次、并发任务和缓存必须有界。接近容量时采用入口限流、排队上限、背压和负载丢弃，不能让请求无限积压直至内存耗尽。
- 定义功能降级顺序：排行榜、推荐、分析、部分聊天等非核心能力可降级；认证、资产、交易和持久化无法确认时应失败关闭，不能冒险返回成功。
- 外部依赖持续失败时使用熔断和隔离，恢复后逐步放量。一个慢接口不能占满所有线程、连接或回调槽位。
- `onReadyForLogin` 只有在脚本、关键配置、数据库与必要依赖真正就绪后才开放登录；启动未完成或版本不兼容时明确拒绝流量。
- 组件故障恢复期间限制并发恢复量并监控积压，避免 Base Entity 批量恢复、玩家集中重连和缓存预热同时压垮数据库。
- 高可用、备份和灾难恢复是三件事：副本不能代替历史备份，自动重启不能代替跨故障域恢复，必须分别设计和验证。
- 自动/人工故障切换必须先隔离旧写入者，再让新写入者接管；无法确认旧实例已失去写权限时不得形成双写或宣称切换成功。
- 灾备恢复同时覆盖数据库、游戏脚本、配置/数据表、密钥/证书和版本映射；只有数据而缺少匹配运行环境的备份不可视为可恢复。
- 使用稳定的服务器时间和唯一业务 ID。处理系统时钟跳变、跨时区和夏令时；活动与过期逻辑存储统一时间基准，不信任客户端时间。
- 定期演练单组件故障、网络分区、数据库降速、磁盘将满、日志服务不可用和重连风暴。每个演练都验证数据一致性、降级效果和恢复时间。

## Timer、异步与热更

- Timer 回调使用 Entity/Component 方法或模块顶层函数；不用 lambda、闭包和局部函数，以保证热更可以稳定重新定位。
- Timer、异步回调执行前重新确认实体仍有效、状态仍匹配。销毁实体时清理不再需要的 Timer 和挂起操作。
- `KBEngine.reloadScript(False)` 只用于已加载 Python 模块的逻辑修复。热更前后检查 changed files、失败日志与 `timersKeptOld`。
- 生产热更不得修改 `.def`、`types.xml`、持久化字段、插件集合或跨进程协议，不在 `onReload` 中迁移数据。
- 热修经过开发与灰度验证后逐进程执行，并安排后续滚动重启恢复干净状态。

## 编码约定

- 遵循生成资产库现有的 Python 风格和 KBEngine 生命周期命名，不进行无关格式化或重构。
- 用最少必要代码完成当前需求；不为一次性功能增加抽象层、事件总线、插件或配置系统。
- 业务规则与网络回调分离：RPC 负责鉴权、校验和调用领域逻辑，领域逻辑返回明确结果。
- 配置表数据在加载阶段验证引用、范围和唯一性。运行期逻辑不默默吞掉非法配置，也不随意使用默认值掩盖错误。
- 错误处理区分玩家输入错误、可重试基础设施错误和程序缺陷；不要使用裸 `except`。
- 日志包含必要上下文与稳定事件名，但避免在高频路径逐 Tick 打日志。
- 不新增第三方依赖，除非先说明原生方案不足、部署影响和替代方案并取得确认。

## 开发阶段

- 一个开发任务应有：需求边界、验收示例、数据权威、Entity/Base/Cell 归属、失败行为和验证计划。
- 优先在本地最小拓扑完成纵向闭环：登录、创建/加载角色、进入 Space、执行玩法、持久化、退出和重新登录验证。
- 配置只在资产库 `res/server/kbengine.xml` 覆盖引擎默认值，不直接修改 `kbe/res/server/kbengine_defaults.xml`。环境差异通过受控配置或部署注入处理。
- 开发环境使用 `publish.state=0`；生产发布使用明确的非调试状态。每次可部署变更同步更新并记录 `publish.script_version`，版本不得含糊地复用。
- 新功能默认支持关闭或隔离，尤其是经济、跨服、活动和外部服务功能。功能开关必须有安全默认值、权限保护和清理计划，不能永久堆积。
- 配置表、掉落表、数值公式和协议生成应可重复执行；生成物来源明确，不依赖个人机器上的未提交文件。
- 提交保持小而可审查，协议、迁移、业务逻辑和测试的关系清楚；不混入无关格式化、引擎升级或临时运维改动。

## 测试阶段

- 纯业务规则优先写成不依赖运行中 KBE 进程的单元测试。
- Entity/RPC 功能至少验证：正常请求、越权请求、边界值、重复请求、乱序/超时、断线重连和实体销毁中的回调。
- 经济系统测试必须断言资产守恒、幂等和失败无部分提交。
- 战斗系统使用固定输入或固定随机种子验证结算顺序、边界和死亡流程。
- 跨 Base/Cell、Space 切换和 SDK 协议变更使用集成测试；并用 `bots/` 做登录、移动、战斗、重连和并发压力场景。
- 性能相关修改记录场景、在线实体数、请求/Tick 频率、CPU、内存、延迟和错误率，不能以“本机感觉正常”作为结论。
- 测试按层执行：静态/配置校验、单元测试、组件测试、Base/Cell 集成测试、客户端协议联调、Bots 场景、压力与耐久测试、故障恢复测试。
- 每次协议变更至少用一个实际目标 SDK 完成登录、实体创建、属性同步及 Base/Cell RPC 往返；多种 SDK 同时在用时，维护明确的服务端—客户端兼容矩阵。
- 压测模型必须接近真实分布，包括登录峰值、同屏密度、技能频率、聊天/交易、数据库写入和断线重连风暴；分别记录平均值、P95/P99 和错误率。
- 进行长时间耐久测试，观察 Entity/Timer 数量、内存、句柄、队列积压、数据库连接与日志增长，排查缓慢泄漏。
- 故障注入至少考虑 BaseApp/CellApp 异常、数据库短暂不可用、外部接口超时、重复消息、网络抖动和安全停服恢复，并检查数据是否重复或丢失。
- 对公开 RPC 做边界、超长、畸形、乱序、高频、重放和无权限调用测试；确认服务端拒绝请求且进程、内存和日志量保持稳定。
- 对登录、移动、战斗、聊天、交易和外部回调进行滥用/作弊场景测试，验证限流与审计证据，不只验证正常玩家路径。
- 对角色生命周期、交易、支付和跨服迁移状态机做转换覆盖测试，验证非法转换、超时、崩溃重启和重复回调不能产生双角色或重复资产。
- 使用属性/不变量测试验证资产守恒、HP/资源范围、唯一归属、同一角色单写者等规则；能重放的战斗或经济缺陷保存输入和随机种子作为回归用例。
- schema/配置迁移使用接近生产规模和分布的数据验证耗时、锁、暂停续跑、重复执行、前后核对以及回滚/向前修复路径。
- 保存代表性场景的性能基线。同等硬件与负载下若 P95/P99、CPU、内存、网络或数据库指标明显退化，必须解释并确认后才能发布。
- 发布候选版本冻结后只允许修复阻断问题；任何修复都重新执行受影响测试，不沿用旧测试结论。
- 只运行与改动风险相称的最小验证集；若仓库暂时没有自动测试框架，先补最小可复现脚本，不伪造测试通过。

## 上线阶段

### 发布准备

- 发布必须有唯一版本和变更清单，关联代码提交、脚本版本、配置版本、数据表版本、schema 版本、SDK 目标及已知风险。
- 使用同一发布候选产物完成预发布验证后再上线；禁止在生产机器临时编辑 Python、XML、配置表或 SDK。
- 上线前确认容量余量、端口与防火墙、数据库连接池、磁盘空间、日志轮转、时间同步、外部依赖和告警接收人。
- 上线前执行安全配置审计：无示例/默认凭据，无调试或管理端口公网暴露，外部通道加密与窗口限制生效，数据库运行账号为最小权限，证书与密钥在有效期内。
- 对 schema 或数据迁移先在生产数据副本上演练，记录耗时、锁影响、失败恢复和回滚限制。上线前完成可验证备份，并抽样恢复确认备份可用。
- 确认服务端版本与允许登录的客户端版本范围。不兼容协议必须阻止错误版本进入，而不是让其在游戏中随机失败。
- 准备发布步骤、观察窗口、负责人、停止条件和回滚步骤。没有可执行回滚方案的高风险发布不得开始。
- 高风险版本在发布前完成一次停服、恢复或回滚演练；演练未覆盖的步骤必须明确标注并由负责人接受风险。

### 发布执行

- 优先灰度或分批发布：内部服/预发布、少量实例或少量玩家、逐步扩大；每一步达到观察条件后才继续。
- 发布期间冻结无关变更并保存操作时间线。每一步记录执行人、命令/动作、开始结束时间和结果。
- 只有纯 Python 逻辑修复可以按热更规则发布；协议、Entity 定义、持久化结构和插件变化走维护窗口与完整重启。
- 停服使用 `safe_kill`，等待实体写库和组件安全退出。设置合理超时，但不得因等待较久就直接强杀；异常时先判断数据风险并升级处理。
- 启动后依次验证组件健康、登录、加载角色、进入 Space、核心玩法、写库、重连和外部接口，再开放全部流量。

### 发布观察与回滚

- 观察登录成功率、在线数、BaseApp/CellApp 负载、Tick/事件延迟、RPC 错误、数据库延迟与错误、队列积压、内存、异常日志和关键经济指标。
- 出现数据破坏、资产异常、持续错误率上升、延迟超过门槛或组件反复崩溃时停止扩大发布并执行预案。
- 代码回滚不代表数据自动回滚。涉及迁移或新格式时，按预先定义的向前修复、兼容读取或数据补偿方案处理。
- 发布结束后记录结论、实际指标、异常、回滚/补救动作和遗留事项，不能只写“上线成功”。

## 维护阶段

### 可观测性与告警

- 使用 logger、Watcher/WebConsole 等能力观察各组件，但生产控制台和 Python Console 必须限制网络访问与权限。
- 为登录、角色加载、Space 切换、战斗结算、资产流水、写库和外部调用建立稳定事件与指标，能够按版本、组件和错误码定位。
- 跨 Entity、支付、交易和后台任务使用可贯穿调用链的请求/业务流水号，使日志、指标和审计记录能够关联，但不得把敏感 Token 当作追踪 ID。
- 仪表盘同时覆盖技术健康与游戏健康：可用率/延迟/错误/饱和度，以及登录漏斗、在线分布、货币产销、支付发货积压、活动参与和异常封禁。
- SLO 告警关注玩家影响并使用错误预算管理发布速度；底层指标用于定位，不为每个瞬时波动叫醒值班人员。
- 告警必须有阈值、持续时间、严重级别、负责人和处理手册。只收集无人处理的指标不算完成监控。
- 日志设置分级、轮转、保留与脱敏；高频成功路径使用聚合指标，避免日志风暴耗尽磁盘或拖慢组件。

### 日常维护

- 定期验证数据库备份、恢复时间与恢复点目标；恢复演练使用隔离环境并记录结果，确认实测 RPO/RTO 达标。
- 定期审计异常货币产出/消耗、稀有物品、交易、邮件附件、管理命令和充值订单，确保能追溯每次资产变化。
- 定期审计公网端口、管理面账号、密钥权限、依赖漏洞、异常登录/RPC 和封禁规则；发现泄露凭据时先吊销轮换，再调查影响。
- 容量管理基于趋势和活动峰值预估，提前验证扩容、分片与负载均衡，不等资源耗尽后临时处理。
- 清理活动、功能开关、兼容代码和临时补偿逻辑必须作为有期限的维护任务；删除前确认在线对象、存量数据和旧客户端不再依赖。
- 依赖与引擎升级独立立项，先读更新说明并在兼容环境完整回归，不与普通玩法上线捆绑。
- 定期比较实际容量、云/机器/带宽/存储成本与预算。优化先针对测得的主要成本，不以牺牲资产正确性、可恢复性或安全为代价。
- 运行手册至少覆盖：启动/安全停服、扩缩容、组件故障、数据库故障、证书轮换、磁盘将满、支付积压、资产异常和回滚；实际操作变化后同步更新。

### 事故与热修

- 事故处理优先级是保护玩家数据、限制影响范围、恢复服务、再查根因；不得为了快速恢复而跳过数据风险判断。
- 建立事件时间线并保留相关版本、配置、日志、指标和操作记录。沟通中区分已确认事实、推测和下一步动作。
- 热修保持最小改动，先复现并验证；生产执行前确认热更边界、备份、停止条件和回退方式。
- 事故后完成无责复盘：根因、触发条件、为何未提前发现、恢复过程以及可验证的预防措施；修复项要有负责人和期限。

## 停运与系统下线阶段

- 在商业运营前就定义区服合并、角色迁移和最终停运能力，不到关服时才发现账号、支付与角色数据无法安全处理。
- 停运按发行地区与平台要求提前通知，先停止新注册/充值和未来活动，再处理未完成订单、退款/补偿、玩家数据导出或删除请求。
- 最终停服使用安全关闭并核对在线实体写库、支付发货队列、资产流水、批处理和外部回调；保留经验证的最终备份与校验值。
- 按保留策略归档审计、支付和必要玩家数据，到期后可证明地删除。备份、日志、对象存储和第三方副本都纳入删除范围。
- 撤销域名、证书、API Key、数据库账号、管理账号、开放端口、定时任务和第三方回调，避免废弃系统成为攻击入口或继续产生费用。
- 记录最终版本、数据位置、恢复所需依赖、负责人和法律保留期限；完成停运复盘并关闭监控、值班与供应商资源。

## 提交前完成标准

- 行为满足已确认的完成标准，正常与异常测试均通过。
- 服务端仍是所有关键结果的唯一权威，客户端无法通过伪造或重放获利。
- Entity 生命周期、Base/Cell 归属、持久化、失败恢复和并发语义已经明确。
- 涉及角色、交易、支付、活动或跨服流程时，状态机、单一写入者、幂等号、超时和补偿路径完整。
- 若协议发生变化，实体定义、迁移说明、目标 SDK 生成、兼容矩阵和客户端协作事项齐全。
- 发布版本、配置、数据表、schema 与 SDK 可互相追溯；上线步骤、观察门槛和回滚方案齐全。
- 新功能具备必要指标、日志、告警或审计记录，维护人员能够发现并定位异常。
- 架构决定、SLO/RPO/RTO、责任人和运行手册按本次影响更新；没有只存在于个人记忆中的关键操作。
- 安全检查、滥用测试、性能基线和容量评估满足本次风险等级，不存在默认凭据、无界队列、阻塞主循环或公网管理端口。
- 没有敏感信息、调试后门、临时日志、无效导入或本次改动产生的死代码。
- 汇报改动文件、核心设计、实际执行的测试及结果；未执行的验证和已知风险必须明确列出。

## 禁止擅自执行

以下操作必须先说明影响范围和恢复方案，并获得明确确认：

- 删除或批量迁移玩家数据、修改数据库结构、全服批量发放或扣除资产。
- 修改生产配置、密钥、权限、端口、网络策略或调用生产环境 API。
- 强制停服、跳过安全关闭、跳过迁移/校验步骤或把不兼容 schema 当作热更发布。
- 修改 KBEngine 引擎核心、更新核心依赖、改写 Git 历史或强制推送。
